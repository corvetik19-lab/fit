# Light Compact Mobile Brief для `fit`

## Цель

Перевести `fit` в светлый, компактный, интуитивный стиль, который ощущается как коммерчески сильное
fitness-приложение для телефона, а не как web-dashboard с крупными блоками.

## Визуальная теза

- Светлый холст, тёплые нейтрали, чистый графитовый текст.
- Один фирменный акцент на базе сине-бирюзового логотипа.
- Никаких oversized hero-кнопок и «промо-блоков».
- Один главный сценарий на экран, всё вторичное сворачивается или уходит ниже.

## Принципы

- Mobile first, Android/PWA first.
- Карточка используется только там, где она действительно отделяет сценарий.
- Большие вертикальные отступы и длинные explainers сокращаются.
- Текст короткий, утилитарный, на чистом русском.
- Верхний бар и нижняя навигация выглядят как продуктовый shell, а не как декоративная рамка.

## Базовый контракт

- Фон светлый, не серо-грязный и не pure white.
- Основные поверхности почти белые, вторичные — слегка тёплые.
- Кнопки рабочей высоты `40-44px`, а не лендинговые CTA.
- Радиусы умеренные.
- Метрики и списки плотные, быстро сканируются большим пальцем.

## Приоритет экранов

1. `Nutrition` как эталон нового utility-стиля.
2. `Dashboard` и `Workouts`.
3. `Workout day focus-mode`.
4. `AI`, `History`, `Settings`.
5. `Admin`.

## Что считается успехом

- На `360 / 390 / 430px` интерфейс выглядит как настоящее мобильное приложение.
- Нет широких растянутых блоков, которые занимают экран без пользы.
- Основные действия видны сразу.
- На `/nutrition` пользователь за 1-2 взгляда понимает:
  - сколько уже съедено;
  - что можно сделать дальше;
  - где фото, штрихкод и лог дня.

## Обновление 2026-04-15

- Workout day и focus-mode теперь считаются опорным mobile-экраном: меньше hero, меньше крупных панелей, больше плотных рабочих состояний и быстрых числовых действий.
- AI, History и Settings не должны визуально распухать в длинные секции из крупных карточек. Для этих экранов целевой паттерн — короткий summary, затем компактный utility content.
- Любой следующий редизайн этих экранов нужно проверять вопросом: помогает ли это быстрее действовать на телефоне, а не только выглядит ли это эффектно.
- Admin тоже входит в этот контракт: health, queue, bulk actions и карточка пользователя должны выглядеть как светлый operator UI с плотной иерархией, а не как тяжёлый дашборд из крупных тёмных панелей.

## Обновление 2026-04-15 — mobile regression lock

- Финальный mobile acceptance для active light compact UI зафиксирован на трёх ширинах: `360`, `390`, `430`.
- Для каждого нового визуального tranche по user/admin shell обязательно сохранять:
  - отсутствие horizontal overflow;
  - drawer на весь viewport;
  - читаемый compact focus-header в workout focus-mode;
  - отсутствие assistant FAB на admin mobile surfaces.
- Если дизайн визуально красивый, но ломает один из этих четырёх пунктов, он считается неинтегрируемым для production mobile-контракта.
