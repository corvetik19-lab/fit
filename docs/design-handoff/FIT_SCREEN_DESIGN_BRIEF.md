# Полный экранный дизайн-бриф `fit`

## 1. Что это за продукт

`fit` — это web-first fitness-платформа с PWA-подачей, в которой пользователь:

- ведёт тренировки;
- фиксирует питание;
- получает рекомендации от AI-коуча;
- управляет доступом, подпиской и личными данными;
- при необходимости работает через установленную PWA или Android/TWA-обёртку.

Продукт не должен ощущаться как “таблица для спорта”. Он должен ощущаться как:

- персональный инструмент контроля;
- спокойный, уверенный фитнес-интерфейс;
- продукт, в котором важны фокус, прогресс, дисциплина и ясность;
- mobile-first система, пригодная для использования прямо в зале, на кухне и на ходу.

## 2. Цель этого документа

Этот документ нужен, чтобы дизайнер смог:

- спроектировать новый визуальный язык приложения;
- понять реальные экраны и их сценарии;
- не сломать продуктовые потоки;
- отдать дизайн так, чтобы команда могла интегрировать его в текущую кодовую базу без переписывания бизнес-логики.

Документ описывает не только “как экран выглядит”, но и:

- зачем он нужен;
- что на нём главное;
- какие состояния обязательны;
- какие ограничения нельзя нарушать.

## 3. Общий визуальный курс

### 3.1. Общее ощущение продукта

Нужен смелый, современный, премиальный фитнес-язык.

Желаемое ощущение:

- спортивный;
- дорогой по ощущению;
- спокойный;
- собранный;
- без визуального мусора;
- без кислотного “спортпита” и без медицинской стерильности.

### 3.2. Чего хочется добиться

- На мобильном всё должно читаться за секунды.
- Главный экран не должен разваливаться на много равнозначных карточек.
- У каждого экрана должна быть доминанта.
- Действия должны быть понятны без лишнего текста.
- Статусы должны считываться мгновенно.
- Визуально продукт должен отличаться от обычных SaaS-панелей.

### 3.3. Чего не нужно

- случайные одинаковые белые карточки по всей странице;
- декоративность без пользы;
- тяжёлые навигационные панели, которые мешают контенту;
- мелкие нечитабельные фильтры и чипы;
- “AI-slop” композиции без ясной иерархии;
- перегруженный dark cyber-стиль только ради “технологичности”.

### 3.4. Product invariants

Дизайн может меняться смело, но нельзя ломать:

- route map приложения;
- шаговый workout flow;
- focus-mode тренировки;
- AI как chat-first сценарий;
- photo capture, barcode scan и Open Food Facts flow в питании;
- структуру `profile / billing / data` в настройках;
- операторскую природу admin-интерфейса;
- текущие состояния success / error / degraded / locked / empty.

## 4. Глобальные UX-принципы

### 4.1. Mobile/PWA first

Мобильный сценарий — главный.

Это означает:

- layout проектируется сначала под `360-430px`;
- primary CTA доступны большим пальцем;
- длинные страницы дробятся на секции;
- ключевые сценарии не должны зависеть от desktop-hover;
- фиксированные панели допускаются только там, где они реально помогают.

### 4.2. Desktop не копия mobile

Desktop должен быть:

- просторнее;
- информативнее;
- плотнее по контенту;
- но с той же визуальной системой.

Desktop не должен просто “растягивать” мобильные карточки.

### 4.3. Единая логика workspace-экранов

Для больших экранов приложения нужен единый паттерн:

- верхняя hero/summary-зона;
- переключение логических секций;
- одна доминирующая секция в фокусе;
- вторичные блоки вокруг, а не бесконечная лента.

### 4.4. Состояния обязательны

Для каждого важного экрана дизайнер должен отдать не только default state, но и:

- loading;
- empty;
- error;
- success;
- degraded;
- locked/read-only, если применимо;
- mobile collapsed/expanded, если применимо.

## 6. Общие рекомендации по shell и навигации

### 6.1. App shell

Во всём приложении нужен единый app shell:

- узнаваемый;
- компактный;
- пригодный для PWA;
- безопасный по safe area;
- не перекрывающий основной контент.

На mobile:

- короткий header;
- аккуратный drawer;
- без навязчивых фиксированных плашек;
- контент важнее shell.

На desktop:

- верхняя навигация должна быть ясной;
- не должно быть ощущения “плавающей панели поверх всего”.

### 6.2. Section navigation

Для больших экранов продукта section-navigation уже встроена в логику.

Дизайнеру нужно сохранить сам принцип:

- секции есть;
- секции читаемы;
- активная секция явно выделяется;
- на mobile это не превращается в длинную простыню табов.

### 6.3. CTA

Должно быть чёткое разделение:

- primary action;
- secondary action;
- destructive action;
- utility action.

Сейчас в продукте много состояний типа:

- сохранить;
- завершить;
- подтвердить;
- открыть;
- обновить;
- отменить;
- редактировать;
- сбросить.

Дизайн должен сделать их мгновенно различимыми.

## 7. Экран 1: `/` — старт и вход

### Кто видит

- неавторизованный пользователь;
- новый пользователь;
- возвращающийся пользователь до входа.

### Главная задача

- быстро понять, что это за продукт;
- войти или зарегистрироваться;
- без перегруза начать путь в приложение.

### Главный эмоциональный режим

- уверенный старт;
- ощущение “входа в серьёзный продукт”, а не “регистрационной формы”.

### Что обязательно должно быть

- hero-блок с коротким смыслом продукта;
- переключение `вход / регистрация`;
- форма;
- понятные error/notice состояния;
- на mobile форма должна быть приоритетом;
- на desktop можно сильнее показать brand/hero.

### Приоритет блоков

1. Hero и основное обещание продукта
2. Форма входа/регистрации
3. Ошибка или notice
4. Вторичные текстовые подсказки

### Mobile

- форма должна помещаться без ощущения перегруза;
- hero может быть короче, чем на desktop;
- нужно ощущение “быстро войти и продолжить”.

### Desktop

- можно делать split-layout;
- визуально hero может быть сильнее и эмоциональнее;
- форма всё равно должна оставаться читаемой и компактной.

### Состояния

- default sign-in;
- sign-up;
- pending;
- ошибка входа;
- notice после регистрации.

### Что нельзя ломать

- режимы `Вход` и `Регистрация` должны остаться;
- форма должна поддерживать имя только для регистрации;
- после входа пользователь уходит в `/dashboard`, после регистрации с сессией — в `/onboarding`.

## 8. Экран 2: `/onboarding` — анкета и стартовый профиль

### Кто видит

- пользователь после входа, если базовый профиль ещё не заполнен.

### Главная задача

- быстро заполнить параметры, от которых зависят тренировки, питание и AI.

### Что это по сути

Это не “длинная форма ради формы”, а создание стартового контекста.

### Что обязательно должно быть

- понятное объяснение, зачем нужна анкета;
- ощущение движения вперёд;
- блоки:
  - базовые данные;
  - физические параметры;
  - уровень;
  - цель;
  - тренировочная частота;
  - оборудование;
  - ограничения;
  - пищевые предпочтения.

### Важные поля

- имя;
- возраст;
- пол;
- рост;
- вес;
- уровень подготовки;
- цель;
- тренировок в неделю;
- целевой вес;
- оборудование;
- травмы/ограничения;
- пищевые предпочтения.

### Mobile

- форма должна ощущаться пошаговой даже если технически это один экран;
- нужны визуальные группы полей;
- primary CTA всегда должен быть понятен.

### Desktop

- допустим более широкий form-grid;
- но смысловые группы должны быть видны лучше, чем просто таблица инпутов.

### Состояния

- default;
- validation error;
- pending save;
- success/transition в продукт.

### Что нельзя ломать

- этот экран должен оставаться быстрым стартом, а не длинным опросником;
- обязательные поля должны оставаться очевидными;
- после успеха пользователь попадает в `/dashboard`.

## 9. Экран 3: `/dashboard` — обзор и статус дня

### Кто видит

- авторизованный пользователь после онбординга.

### Главная задача

- быстро понять текущее состояние прогресса;
- увидеть, что важно прямо сейчас;
- перейти к нужному действию.

### Что это за экран

Это не “страница с графиками”. Это экран статуса дня.

### Текущая логика разделов

- `Сводка`
- `Тренировки`
- `Питание`
- `AI`

### Что обязательно должно быть

- сильный верхний summary/hero;
- KPI-блок;
- переключение разделов;
- акцент на текущем приоритете;
- AI-summary как coach layer, а не как отдельный чужой модуль.

### Обязательные типы информации

- активные недели;
- завершённые тренировки;
- записанные подходы;
- дни питания;
- сравнение периодов;
- workout analytics;
- nutrition analytics;
- AI insights.

### Mobile

- один главный смысл на экран;
- секции должны открываться как понятные контентные поверхности;
- графики не должны превращаться в длинную колонку одинаковых блоков.

### Desktop

- можно показывать больше аналитики одновременно;
- но у экрана всё равно должна быть одна визуальная доминанта.

### Состояния

- normal;
- no data / только начало использования;
- stale snapshot / пересчитано сейчас;
- section switch on mobile;
- degraded analytics source.

### Что нельзя ломать

- section-модель `summary / workouts / nutrition / ai`;
- dashboard должен оставаться верхнеуровневым экраном, а не deep-work screen.

## 10. Экран 4: `/workouts` — программы, недели, упражнения

### Кто видит

- обычный пользователь, управляющий тренировочным контуром.

### Главная задача

- собрать, активировать и понимать свою недельную структуру тренировок;
- управлять библиотекой упражнений;
- открыть конкретный тренировочный день.

### Что обязательно должно быть

- workspace hero;
- ключевые метрики;
- section navigation;
- минимум три смысловых слоя:
  - неделя / программа;
  - шаблоны;
  - библиотека упражнений.

### Текущая продуктовая структура

На экране уже живут:

- weekly program builder;
- active week;
- templates;
- exercise library.

### Как должен ощущаться экран

- как “центр подготовки тренировок”;
- не как админка с формами;
- не как spreadsheet.

### Mobile

- секции должны открываться по очереди;
- список упражнений и builder не должны спорить между собой за внимание;
- активная неделя должна быть выше по приоритету, чем конструктор исторических сущностей.

### Desktop

- можно раскрывать больше редакторских возможностей;
- важно сохранить ясность между:
  - текущей неделей;
  - созданием новой недели;
  - шаблонами;
  - библиотекой.

### Состояния

- нет упражнений;
- нет программ;
- есть draft;
- есть active program;
- archive/history;
- pending create/save.

### Что нельзя ломать

- builder-поток;
- active week;
- templates;
- exercise library;
- переход в `/workouts/day/[dayId]`.

## 11. Экран 5: `/workouts/day/[dayId]` — экран тренировки

### Кто видит

- пользователь во время или перед выполнением тренировочного дня.

### Главная задача

- пройти тренировку последовательно;
- не отвлекаться;
- сохранить факт выполнения;
- видеть шаги, таймер и прогресс.

### Это один из самых критичных экранов продукта

Именно этот экран должен стать главной mobile-поверхностью всего приложения.

### Обязательные части

- day header / workout context;
- focus-mode;
- step strip;
- active exercise card;
- sets / reps / weight / RPE;
- timer;
- save state;
- finish state;
- reset state;
- read-only state после сохранения упражнения;
- edit action после сохранения.

### Логика, которую нужно уважать

- все упражнения видны в step strip сразу;
- будущие шаги закрыты;
- следующий шаг не открывается, пока текущий не сохранён;
- упражнение после сохранения становится read-only;
- есть `Редактировать`, если нужно изменить;
- таймер живёт между сессиями;
- reset реально сбрасывает состояние дня.

### Обычный режим

Нужен контекст:

- день;
- навигация назад;
- сводка;
- общий режим работы с экраном.

### Focus-mode

Это главный сценарий на mobile.

Здесь должно остаться только нужное:

- компактный header;
- таймер;
- прогресс;
- активный шаг;
- быстрые действия.

### Mobile

- активное упражнение должно визуально доминировать;
- шаги должны быть заметны, но не шумны;
- header может быть sticky/collapsible;
- таймер и прогресс должны считываться мгновенно;
- завершённые шаги должны явно отличаться от будущих и активных.

### Desktop

- можно давать больше контекста;
- но focus-mode визуально должен всё равно ощущаться собранным и рабочим.

### Состояния

- planned;
- in progress;
- done;
- active timer;
- paused timer;
- exercise ready to save;
- exercise saved;
- locked/read-only;
- edit mode;
- finish available / finish unavailable;
- reset pending;
- sync offline/online;
- error and notice.

### Что дизайнер обязан показать

- обычный режим;
- focus-mode;
- шаг “текущий”;
- шаг “закрыт”;
- шаг “сохранён”;
- read-only exercise;
- редактирование после сохранения;
- завершение тренировки;
- reset warning / reset result.

### Что нельзя ломать

- step-based структура;
- focus-mode;
- read-only после сохранения;
- таймер;
- последовательность completion logic.

## 12. Экран 6: `/nutrition` — питание, дневной контроль и импорт

### Кто видит

- обычный пользователь в daily nutrition flow.

### Главная задача

- понять, что съедено сегодня;
- добавить еду;
- сохранить приём пищи;
- использовать фото и штрихкод;
- быстро импортировать продукт.

### Что это за экран

Это не “справочник еды”. Это экран дневного контроля.

### Текущая структура секций

- balance
- photo
- log

Внутри трекера:

- targets
- foods
- log
- history

### Что обязательно должно быть

- верхний дневной summary;
- macro progress;
- CTA для добавления еды;
- фото-анализ;
- barcode scan;
- Open Food Facts import;
- food library;
- meal log;
- history/summary.

### Важные сущности

- foods;
- meals;
- recipes;
- meal templates;
- daily summary;
- nutrition targets.

### Mobile

- дневной баланс выше справочников;
- photo/barcode flow должны ощущаться быстрыми;
- импорт продукта не должен быть buried внутри формы;
- meal log должен быть удобен для частого использования.

### Desktop

- можно показывать больше параллельных блоков;
- но приоритет daily control остаётся первым.

### Состояния

- today empty;
- есть summary;
- цели заданы / не заданы;
- barcode found;
- barcode not found;
- photo analysis success / fail;
- import preview;
- food editing;
- meal logged;
- history present / absent.

### Что нельзя ломать

- фото еды;
- штрихкод;
- Open Food Facts import;
- targets / foods / log / history;
- daily-control nature of the screen.

## 13. Экран 7: `/ai` — AI-коуч

### Кто видит

- пользователь с доступом к AI-функциям.

### Главная задача

- задать вопрос;
- получить анализ;
- получить предложение;
- подтвердить и применить его.

### Что это за экран

Это полноценный chat workspace, а не просто чат-виджет.

### Текущая структура

- transcript;
- composer;
- history;
- context;
- plans;
- assistant flow:
  - request
  - analysis
  - proposal
  - approve
  - apply

### Что обязательно должно быть

- доминирующий transcript;
- удобный composer;
- боковой или секционный доступ к history/context/plans;
- явное отображение текущего шага AI flow;
- states доступа к AI и meal-photo;
- user-facing, а не технический язык.

### Mobile

- composer удобен для большого пальца;
- transcript — главный фокус;
- вторичные разделы не должны ломать основной chat;
- история и планы должны открываться как понятные вторичные поверхности.

### Desktop

- можно позволить более богатую компоновку;
- но transcript должен оставаться главным.

### Состояния

- empty first use;
- active conversation;
- no access;
- AI unavailable;
- proposal draft;
- approved;
- applied;
- image attached;
- history empty;
- context degraded.

### Что нельзя ломать

- chat-first модель;
- history / context / plans;
- proposal-first flow;
- meal photo attach;
- AI access gating.

## 14. Экран 8: `/history` — история программ и операций

### Кто видит

- обычный пользователь, который хочет посмотреть прошлые результаты и события.

### Главная задача

- понять, что уже было сделано;
- увидеть прошлые программы;
- увидеть AI proposals;
- увидеть историю действий по данным.

### Что обязательно должно быть

- history hero;
- секционность;
- разделение на типы истории;
- не ощущаться как “сырой журнал событий”.

### Логические слои

- программы и недели;
- AI proposals;
- privacy/data events.

### Mobile

- история должна быть разложена по разделам;

- есть программы;
- есть proposals;
- есть privacy events;
- mixed history.

### Что нельзя ломать

## 23. Ключевой принцип интегрируемости

Хороший редизайн для `fit` — это не только красивый макет.

Он должен:

- усиливать продукт;
- сохранять реальные route-flow и states;
- быть совместимым с текущими workout, nutrition, AI и admin сценариями;
- не требовать переписывать backend-контракты ради красивого экрана.

Если между “очень эффектно” и “реально интегрируется” есть конфликт, приоритет у варианта, который:

- сохраняет продуктовую логику;
- даёт сильную mobile-first композицию;
- улучшает понятность;
- можно внедрить в текущую архитектуру без ломки основных flows.

## 24. Актуально внедрённый stitch-язык

На текущем этапе в продукт уже внедрён новый visual baseline по референсу `stitch_`. Дизайнеру и разработчику важно опираться не на старый premium-план, а на реально внедрённую систему.

Что уже считается каноническим:

- `Lexend + Manrope` как типографическая пара;
- светлый editorial canvas с electric blue как главным акцентом;
- `card card--hero` как базовый hero-сurface;
- `surface-panel` для вторичных смысловых блоков;
- `section-chip` и `toggle-chip` для section navigation и visibility controls;
- чистый mobile-first shell с drawer и bottom navigation без лишних floating-панелей.

Ключевые реализованные экраны нового языка:

- `Dashboard`
- `Workout day` и mobile focus-mode
- `Nutrition`
- `AI`
- remaining user screens: `/`, `/onboarding`, `/workouts`, `/history`, `/settings`, `/billing/cloudpayments`, `/suspended`

## 25. Актуализация стартового экрана `/`

- На текущем внедрённом варианте стартовый маршрут `/` больше не использует split-layout с левой промо-колонкой.
- Для дизайнера и разработчика это теперь жёсткий контракт: на входном экране остаются только `brand chip`, заголовок, переключение `Вход / Регистрация` и сама форма.
- Социальные входы `Google` и `Apple` исключены из продукта и не должны возвращаться в макеты без отдельного product-решения.
- Если экран будет перерабатываться повторно, его нужно развивать как `single-column auth entry`, а не как маркетинговый landing с отдельной левой панелью.
- `Admin`, `/admin/users`, `/admin/users/[id]`

Если дизайнер делает новые варианты или уточнения, они должны:

- усиливать именно этот stitch-style baseline, а не возвращать проект к старой mixed-style подаче;
- сохранять mobile-first композицию;
- уважать уже встроенные состояния `degraded`, `locked`, `empty`, `error`, `success`;
- быть пригодными для интеграции в текущие shared primitives, а не требовать параллельного второго UI-слоя.
